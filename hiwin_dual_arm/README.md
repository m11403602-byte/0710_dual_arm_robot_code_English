# <p align="center">hiwin_dual_arm — Dual-Arm Robot Description + MoveIt2 Configuration Meta-Package</p>

<p align="justify">
This package is generated by the MoveIt Setup Assistant and is the <b> robot-description and MoveIt2 configuration meta-package </b> for the HIWIN dual-arm manipulator (Arm A <code>big</code> = RA610, Arm B <code>small</code> = RA605). It <b> contains no collision-avoidance algorithm </b> itself; its role is that of the "chassis": the robot model (URDF/SRDF), the ros2_control controllers, and the wiring of the six custom collision-avoidance planners ( <code>dual_arm_{alm,lag}_{newton,cg,gd}_planner</code> , see the workspace-root <a href="../README.md">README.md</a> ) to MoveIt2 are all connected through this package's <code>config/*_planning.yaml</code> and <code>launch/*.py</code> .
</p>

<p align="justify">
This document explains what is inside this package, how the six planner plugins are currently wired in (and which step is still missing), and the complete startup procedure.
</p>

---

### SRDF Planning Groups ( <code>config/dual_hiwin.srdf</code> )

```
group "big"      : world -> big_tool0    (Arm A RA610, joint-name prefix big_joint_)
group "small"    : world -> small_tool0  (Arm B RA605, joint-name prefix small_joint_)
group "Dual_arm" : big + small           (merged dual-arm group — all six collision-avoidance planners solve for this group)
```

### Controller Mapping ( <code>config/moveit_controllers.yaml</code> + <code>ros2_controllers.yaml</code> )

<p align="justify">
The robot is split into <b> two independent namespaces </b> ( <code>small_arm/</code> , <code>big_arm/</code> ), corresponding to the common real-deployment scenario in which the two arms each run their own controller — or even each run on a separate computer:
</p>

<ul>
  <li align="justify" style="margin-bottom: 8px;"><code>moveit_controllers.yaml</code> : the actions MoveIt invokes are <code>/small_arm/joint_trajectory_controller/follow_joint_trajectory</code> and <code>/big_arm/joint_trajectory_controller/follow_joint_trajectory</code> .</li>
  <li align="justify" style="margin-bottom: 8px;"><code>srcipt/joint_state_merger.py</code> : subscribes to <code>/small_arm/joint_states</code> and <code>/big_arm/joint_states</code> and merges them into a single <code>/joint_states</code> (12 axes), because MoveIt / RViz must see the <b> complete, merged </b> joint state of both arms; it also overwrites the timestamp with the host clock, preventing MoveIt's 1-second freshness threshold from deeming the data stale on account of cross-subnet / cross-machine clock skew.</li>
</ul>

> <div align="justify">⚠️ The controller naming in <code>config/ros2_controllers.yaml</code> ( <code>big_controller</code> / <code>small_controller</code> , without a namespace prefix) is currently <b> not entirely consistent </b> with that in <code>moveit_controllers.yaml</code> ( <code>big_arm/joint_trajectory_controller</code> / <code>small_arm/joint_trajectory_controller</code> ). In an actual deployment (especially when running controller_manager separately on two computers), please confirm that the namespaces / controller names on both sides align, otherwise MoveIt will be unable to find the action server when executing a trajectory.</div>

## 2. How the Six Collision-Avoidance Planner Plugins Are "Currently" Wired

<p align="justify">
Under <code>config/</code> there are already six <code>dual_arm_&lt;model&gt;_&lt;solver&gt;_planning.yaml</code> files, each specifying:
</p>

```yaml
planning_plugin: 'dual_arm_alm_newton_planner/DualArmAlmNewtonPlannerManager'   # varies per file
request_adapters: >-
  default_planner_request_adapters/FixWorkspaceBounds
  ...
danger_threshold: 0.4
...(all tunable parameters of that planner; see each planner package's PARAMETERS.md)
```

<p align="justify">
This follows MoveIt2's standard filename convention of "one <code>&lt;pipeline_id&gt;_planning.yaml</code> per pipeline" ( <code>pipeline_id</code> = the filename with <code>_planning.yaml</code> removed, e.g., <code>dual_arm_alm_newton_1</code> ).
</p>

<p align="justify">
<b> However, at present neither <code>package.xml</code> nor any of the <code>launch/*.py</code> files actually wires these six pipelines into <code>move_group</code> </b> — when <code>move_group.launch.py</code> / <code>demo.launch.py</code> / <code>brain.launch.py</code> call <code>MoveItConfigsBuilder(...).to_moveit_configs()</code> , they do <b> not </b> call <code>.planning_pipelines(...)</code> , so the planning pipelines actually loaded are only MoveIt2's built-in defaults (OMPL, and possibly Pilz depending on the <code>moveit_configs_utils</code> version); the six custom planners <b> are not yet in effect </b> .
</p>

<p align="justify">
To make the plugins genuinely recognized by <code>move_group</code> and selectable from the "Planning Pipeline" drop-down in RViz, the following two steps are required (see the next section).
</p>

## 3. How to Actually Apply the Planner Plugins

### Step 1: Declare Dependencies in <code>package.xml</code>

<p align="justify">
In <code>package.xml</code> , add the <code>exec_depend</code> for the six planner packages (so that colcon knows the build order, and so that ament's plugin index can be found by <code>pluginlib</code> within the same workspace):
</p>

```xml
<exec_depend>dual_arm_alm_newton_planner</exec_depend>
<exec_depend>dual_arm_alm_cg_planner</exec_depend>
<exec_depend>dual_arm_alm_gd_planner</exec_depend>
<exec_depend>dual_arm_lag_newton_planner</exec_depend>
<exec_depend>dual_arm_lag_cg_planner</exec_depend>
<exec_depend>dual_arm_lag_gd_planner</exec_depend>
```

### Step 2: Declare <code>planning_pipelines</code> in the launch File

<p align="justify">
Modify <code>move_group.launch.py</code> (and also <code>demo.launch.py</code> / <code>brain.launch.py</code> if you wish to select planners there), calling <code>.planning_pipelines(...)</code> to list the six custom pipeline ids:
</p>

```python
from moveit_configs_utils import MoveItConfigsBuilder
from moveit_configs_utils.launches import generate_move_group_launch

def generate_launch_description():
    moveit_config = (
        MoveItConfigsBuilder("dual_hiwin", package_name="hiwin_dual_arm")
        .planning_pipelines(
            pipelines=[
                "ompl",                     # keep the built-in OMPL as a fallback
                "dual_arm_alm_newton_1",
                "dual_arm_alm_cg_1",
                "dual_arm_alm_gd_1",
                "dual_arm_lag_newton_1",
                "dual_arm_lag_cg_1",
                "dual_arm_lag_gd_1",
            ],
            default_planning_pipeline="dual_arm_alm_newton_1",   # which one to use by default
        )
        .to_moveit_configs()
    )
    return generate_move_group_launch(moveit_config)
```

<p align="justify">
Every string in the <code>pipelines</code> list must correspond to an existing <code>config/&lt;string&gt;_planning.yaml</code> ; <code>MoveItConfigsBuilder</code> will follow this convention to read the <code>planning_plugin</code> / <code>request_adapters</code> / parameters from the corresponding file and inject them into the <code>move_group</code> node.
</p>

> <div align="justify">Choosing one of the six only requires "selecting one as the <code>default_planning_pipeline</code> " — place all six into the <code>pipelines</code> list, and you can then switch among them in real time from the "Planning Pipeline" drop-down of RViz's MotionPlanning panel, comparing the planning results of all six algorithms without restarting.</div>

### Step 3: Confirm the Pairing of <code>time_optimal</code> and <code>request_adapters</code>

<p align="justify">
The <code>request_adapters</code> list in each of the six yaml files deliberately <b> excludes </b> <code>default_planner_request_adapters/AddTimeOptimalParameterization</code> (because time parameterization is instead handled inside the plugin's <code>solve()</code> ; if <code>time_optimal: false</code> yet the adapter runs TOTG again, it would overwrite the custom equal-interval time settings). To switch to <code>time_optimal: true</code> (TOTG), either of two approaches works — pick one:
</p>

<ul>
  <li align="justify" style="margin-bottom: 8px;">Keep the plugin's built-in TOTG (leaving <code>request_adapters</code> untouched), or</li>
  <li align="justify" style="margin-bottom: 8px;">Revert to letting the adapter perform TOTG, and disable the <code>time_optimal</code>-related logic in that yaml. In general it is recommended to keep the status quo (handling it inside the plugin); there is no need to stack the two.</li>
</ul>

### Step 4: Rebuild

```bash
colcon build --symlink-install --packages-up-to hiwin_dual_arm
source install/setup.bash
```

<p align="justify">
( <code>--packages-up-to hiwin_dual_arm</code> will build the six planner packages along with it, because Step 1 already declared the dependencies in <code>package.xml</code> .)
</p>

## 4. Complete Startup Procedure

### 4.1 Prerequisite: Enter the ROS 2 Humble Environment and Build (distrobox container)

```bash
distrobox enter hiwin-humble-env

# inside the container, first clear any lingering system ROS environment variables to avoid conflicting with humble
unset ROS_VERSION ROS_ROOT AMENT_PREFIX_PATH CMAKE_PREFIX_PATH COLCON_PREFIX_PATH
source /opt/ros/humble/setup.bash

cd ~/0706_hiwim         # workspace root
rm -rf build/ install/ log/     # clean rebuild (only needed if stale cache remains)
colcon build --symlink-install --cmake-args -DCMAKE_PREFIX_PATH=/opt/ros/humble
source install/setup.bash

export ROS_DOMAIN_ID=24   # unify the domain ID when multiple machines/containers share a network
```

### 4.2 Just Checking Whether the Trajectory Looks Abnormal (pure visualization, the fastest)

```bash
ros2 launch hiwin_dual_arm demo.launch.py
```

<p align="justify">
<code>demo.launch.py</code> uses the built-in <code>generate_demo_launch</code> from <code>moveit_configs_utils</code> , which automatically starts, in one go: the virtual TF, <code>robot_state_publisher</code> , fake controllers (fake hardware), <code>move_group</code> , and RViz2. It is well suited to pure planning/visualization debugging and requires neither real hardware nor <code>spawn_controllers</code> .
</p>

> <div align="justify">If you have already wired the six custom pipelines per §3, the RViz MotionPlanning panel can switch planners; if you have not (the current default state), only MoveIt's built-in OMPL is available here.</div>

### 4.3 Component-by-Component Manual Startup (for a real robot / debugging; start each node individually)

<p align="justify">
In sequence, in separate terminals (or chained together in a single launch file via <code>IncludeLaunchDescription</code> , which is what <code>brain.launch.py</code> does):
</p>

```bash
# 1) virtual joint TF (world -> base)
ros2 launch hiwin_dual_arm static_virtual_joint_tfs.launch.py

# 2) Robot State Publisher (computes the 12-axis TF tree)
ros2 launch hiwin_dual_arm rsp.launch.py

# 3) spawn the ros2_control controllers (real/simulated hardware interface + controller_manager)
ros2 launch hiwin_dual_arm spawn_controllers.launch.py

# 4) (if deploying the real dual arms under two separate namespaces) merge joint_states
ros2 run hiwin_dual_arm joint_state_merger.py
#   subscribes to /small_arm/joint_states + /big_arm/joint_states -> publishes the merged /joint_states

# 5) MoveGroup (the MoveIt core brain: receives planning requests, runs the collision-avoidance algorithm, and dispatches trajectories)
ros2 launch hiwin_dual_arm move_group.launch.py

# 6) RViz2 visualization / operator interface
ros2 launch hiwin_dual_arm moveit_rviz.launch.py
```

### 4.4 One-Command Integrated Launch: <code>brain.launch.py</code>

<p align="justify">
<code>brain.launch.py</code> (and the content-identical <code>test1.launch.py</code> ) chains "virtual TF → RSP → MoveGroup → RViz (optional)" into a single launch file:
</p>

```bash
ros2 launch hiwin_dual_arm brain.launch.py            # with RViz
ros2 launch hiwin_dual_arm brain.launch.py use_rviz:=false   # without RViz (run the brain purely in the background)
```

> <div align="justify">⚠️ Note: <code>brain.launch.py</code> currently does <b> not </b> include <code>spawn_controllers.launch.py</code> , nor <code>joint_state_merger</code> . Pure planning/visualization is fine, but to actually dispatch a trajectory for execution (the <code>FollowJointTrajectory</code> action), you must additionally start <code>spawn_controllers.launch.py</code> (and/or the real controller nodes) plus, depending on the deployment, possibly <code>joint_state_merger</code> . To make it a single entry point, you can follow the pattern of §3 and add, within <code>brain.launch.py</code> , <code>spawn_controllers.launch.py</code> via <code>IncludeLaunchDescription</code> together with a <code>Node(package="hiwin_dual_arm", executable="joint_state_merger.py")</code> (this first requires adding the Python-executable install settings in <code>CMakeLists.txt</code> / <code>package.xml</code> , since <code>srcipt/joint_state_merger.py</code> is not yet installed as an executable ROS node via <code>install(PROGRAMS ...)</code> ).</div>

### 4.5 Other Auxiliary Launch Files

```bash
ros2 launch hiwin_dual_arm warehouse_db.launch.py     # start the scene/query database (for storing planning scenes)
ros2 launch hiwin_dual_arm setup_assistant.launch.py  # reopen the Setup Assistant to edit this package
```

## 5. Verifying Whether the Plugins Are Actually Wired In

```bash
# 1. confirm all six planner packages are built and their plugin xml is registered in ament_index
ros2 pkg prefix dual_arm_alm_newton_planner

# 2. watch the log at move_group startup; it prints the list of currently loaded planning pipelines
ros2 launch hiwin_dual_arm move_group.launch.py --show-args   # inspect the arguments
# after startup, check the move_group node's log and search for "Using planning pipeline" / the pipeline names

# 3. RViz MotionPlanning panel -> Planning Pipeline drop-down
#    beyond ompl, the six dual_arm_* options should appear (provided the two-step change in §3 is complete)

# 4. dynamically change a planner parameter (without restarting move_group)
ros2 param set /move_group dual_arm_alm_newton_1.danger_threshold 0.35
```

## 6. What to Adjust in Sync When Changing the Robot / Scene

<ul>
  <li align="justify" style="margin-bottom: 8px;"><code>config/dual_hiwin.urdf.xacro</code> , <code>dual_hiwin.srdf</code> : the robot body, the groups, and the self-collision matrix.</li>
  <li align="justify" style="margin-bottom: 8px;"><code>config/kinematics.yaml</code> , <code>joint_limits.yaml</code> : IK and velocity/acceleration limits (the <code>time_optimal: true</code> TOTG of the six planners requires the acceleration limits here, otherwise it defaults to <code>1 rad/s²</code> and prints a warning).</li>
  <li align="justify" style="margin-bottom: 8px;"><code>config/moveit_controllers.yaml</code> , <code>ros2_controllers.yaml</code> : the controllers/namespaces and action names must match the real robot.</li>
  <li align="justify" style="margin-bottom: 8px;">The <code>joint_prefix_A</code> / <code>joint_prefix_B</code> in the six <code>dual_arm_*_planning.yaml</code> files: these must agree with the joint naming of the SRDF here (currently <code>big_joint_</code> / <code>small_joint_</code> ).</li>
  <li align="justify" style="margin-bottom: 8px;">The robot geometry hard-coded inside the six planner packages (bounding-sphere coordinates and radii, FK, base positions): these are not in this package and must be changed individually in that planner package's <code>newton_solver.cpp</code> / <code>cg_solver.cpp</code> / <code>gd_solver.cpp</code> ; see §8 "Robot Geometry" in each package's <code>PARAMETERS.md</code> .</li>
</ul>
